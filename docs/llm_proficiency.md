# LLM Proficiency: How Claude Was Used in This Project

This document records the scientific use of Claude (claude-sonnet-4-6) in developing
the Chirality Atlas Star Ascidian model. It follows the hackathon prompt-run-test-modify-critique
expectation and documents both successes and failures honestly.

---

## How LLM Was Used Scientifically

Claude was used for five categories of work:

**1. Equation implementation from specifications:**
Given a precise mathematical specification (formula, boundary conditions, numerical scheme),
Claude generated the implementation. Example: the IMEX Gierer-Meinhardt solver was specified
with exact denominator formulas and stability requirements. The generated code was correct
on the first attempt and verified against the expected Turing spot pattern.

**2. Vectorization of force loops:**
Force computations involve pairwise operations over N agents. Claude was asked to vectorize
these into numpy operations with explicit shapes specified. Example: the angular repulsion
force between arm groups required identifying same-center, different-arm pairs -- a
non-trivial masking operation that Claude handled correctly when given the assignment array structure.

**3. Phase sweep infrastructure:**
The parameter sweep loops (run_sweep_A through D in phase_diagram.py) were generated
from a description of the sweep structure, parameter ranges, and output format. The LLM
produced working code that was then profiled and modified to add progress printing and
a speed-vs-accuracy tradeoff (smaller grid for sweeps).

**4. Plot layout and CSS:**
The Streamlit app CSS (assets/style.css) and the matplotlib panel layouts (visualization/panels.py)
were drafted by Claude and then refined by hand to fix alignment issues and remove
overcrowded elements.

**5. Documentation first drafts:**
All docs were generated by Claude and reviewed for accuracy against the actual code.
Some required significant correction (see failure cases below).

## Best Prompt 1: IMEX Gierer-Meinhardt

```
Implement the Gierer-Meinhardt reaction-diffusion system in Fourier space using
an IMEX (implicit-explicit) scheme. Treat diffusion implicitly:

    denom_a = 1 + dt * (Da * k2 + mu_a)
    denom_h = 1 + dt * (Dh * k2 + mu_h)

Treat the nonlinear reaction terms explicitly. In each step:
    1. Compute reaction_a = rho * a^2 / (h * (1 + kappa * a^2)) - mu_a * a + rho_0
    2. Compute reaction_h = rho * a^2 - mu_h * h
    3. Take FFT of a and h
    4. Update: a_hat = (a_hat + dt * fft(reaction_a)) / denom_a
    5. IFFT back to real space, take real part, clip to > 0

The scheme must be unconditionally stable for diffusion. Return activator and inhibitor
histories as (n_snapshots, N, N) arrays.
```

**Why it worked:** The denominator structure was fully specified, leaving no ambiguity
about implicit vs explicit treatment. The clip-to-positive was included in the spec.
Verification: ran at dt=5.0 (ten times default) and confirmed the field did not blow up.

## Best Prompt 2: Vectorized Angular Repulsion

```
Write a numpy function that computes pairwise angular repulsion forces between N agents.
Two arrays are given: assignments (N,) center index, arm_assignments (N,) arm index.

For each pair (i, j) where:
  - assignments[i] == assignments[j]  (same center)
  - arm_assignments[i] != arm_assignments[j]  (different arm)
  - i != j

Compute:
    phi_i = atan2(pos[i] - center, pos[i] - center)   (angle from center)
    dphi = phi_i - phi_j, wrapped to [-pi, pi]
    sigma = 1.5 * 2*pi / n_arms
    if |dphi| < sigma:
        F_mag = -k * (1 - |dphi|/sigma) * sign(dphi)
        F = F_mag * t_hat_i     (tangential unit vector at i)

Return (N, 2) force array. Vectorize the pairwise loop into numpy masks.
```

**Why it worked:** The two-condition mask (same center, different arm) was explicit.
The sign convention (dphi wrapping) was explicit. The tangential vector definition
was left to the LLM, which correctly used t_hat = [-r_hat_y, r_hat_x].
Verification: ran with two agents in the same arm (force should be zero) and two in
adjacent arms (force should push them apart). Both passed.

## What Surprised Us

The LLM was better at IMEX numerical schemes than expected. When the denominator
structure was given explicitly, the code was correct and the scheme was stable.
This saved significant debugging time compared to implementing an explicit Euler scheme
and tuning dt for stability.

The LLM was worse at numpy shape arithmetic than expected. Whenever a formula
mixed `[:, None]` broadcasting with standard 1D arrays in the same expression,
there was a non-trivial risk of a silent shape error. The generated code looked
plausible but produced wrong shapes.

---

## What the LLM Got Wrong

**Swirl score broadcast error:**

The generated swirl_score function in metrics.py contained:

```python
# Generated code -- WRONG
t_hats = np.column_stack([-r_hats[:, 0][:, None] * 0 - r_hats[:, 1], r_hats[:, 0]])
```

The `[:, None]` created a shape (N, 1) column. When subtracted from `r_hats[:, 1]`
(shape N,), numpy broadcast the result to (N, N), not (N, 2). The subsequent dot
product with the (N, 2) velocity array then raised a shape mismatch error.

The error was non-obvious because the expression looks plausible as a tangential vector.
It was caught by reading the traceback carefully and comparing the formula against
the equivalent line in zooid_agents.py (which was written manually and was correct).

**Fix:**
```python
# Correct -- same as zooid_agents.py line for t_hat
t_hats = np.column_stack([-r_hats[:, 1], r_hats[:, 0]])
```

## What Human Judgment Corrected

Beyond the broadcast bug:

1. **phase_diagram.py double-seed error:** BASE_FIELD_PARAMS in the LLM-generated code
   included `seed=42`, but the calling function also passed `seed=seed`. This produced
   "multiple values for keyword argument 'seed'". Fixed by removing seed from the params dict.

2. **n_per_arm passed to wrong function:** The LLM included n_per_arm in BASE_AGENT_PARAMS
   but simulate_zooid_agents does not accept it (it expects n_per_center instead).
   Fixed by filtering the key before unpacking.

3. **CSV writer double-instantiation:** The LLM created two DictWriter instances
   in the same with-block; writerows was called on the second instance which had
   not called writeheader. Fixed by using a single writer instance.

## Why This Was Not Vibe-Coded Blindly

Every generated function was verified with at least three checks before being committed:

1. **Shape check:** `assert output.shape == expected`
2. **Finiteness check:** `assert np.all(np.isfinite(output))`
3. **Physical sanity check:** e.g., same-arm force sums to zero, swirl is near zero
   for omega=0, arm_count is plausible given n_arms parameter.

The smoke test (`scripts/06_final_smoke_test.py`) with 53 checks was run after every
non-trivial change. It caught the n_per_arm bug before any figures were generated.

The phase diagram sweeps were run at reduced grid size (N=32, n_steps=1500) before
committing to full-resolution runs. This caught the double-seed error early.
